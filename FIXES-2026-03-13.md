# CrossX 深度修复报告 (2026-03-13)

## 修复概览

本次修复针对用户反馈的4个核心问题进行了深度检查和完整修复：

### ✅ 问题 1: 天数理解错误（说10天回复20天）

**根本原因：**
1. `extractAgentConstraints` 的正则表达式只匹配阿拉伯数字 `\d{1,2}`，不支持中文数字
2. LLM 系统提示词缺少严格的天数约束指令

**修复内容：**
- **server.js:3487-3502** - 新增中文数字映射表，支持 "十天"、"七天"、"三天" 等20个常用中文数字
- **server.js:2817-2819** - OpenAI 系统提示新增规则 8 和 9：
  - 规则 8: `STRICT DURATION: If user specifies N days, plan EXACTLY N days`
  - 规则 9: `CONTEXT UPDATES: When user changes city/budget/duration mid-conversation, follow the LATEST value only`
- **server.js:3042-3044** - Claude 系统提示同步添加相同规则
- **server.js:3059-3076** - Claude 用户消息改为多语言动态生成（之前硬编码中文）

**测试方法：**
```
输入: "我想去上海玩十天"
预期: 生成恰好10天的行程，不多不少
```

### ✅ 问题 2: 深度推理卡片显示代码而非自然语言

**根本原因：**
1. `buildThinkingNarrative()` 函数被废弃，永远返回空字符串
2. 前端 `renderSmartReplyCard` 的 fallback 逻辑不够健壮，当 `smart.reply` 是 JSON 时会泄露代码片段

**修复内容：**
- **server.js:1934-1951** - 完全重写 `buildThinkingNarrative()` 函数：
  - 从推荐数据生成自然语言推理段落
  - 包含城市、方案数量、最优方案名称、评级、携程数据标识等
  - 支持中英日韩四语
  - 示例输出：`"已为您在上海筛选出 3 个方案。结合携程真实酒店数据，评估了位置、价格、口碑等维度。推荐等级 S 的「上海外滩华尔道夫酒店」综合表现最佳，性价比和用户好评均处于高位。"`

- **web/app.js:7876-7943** - 重构前端推理文本生成逻辑：
  - 优先级：服务端 `smart.thinking` > 干净的 `smart.reply` 文本 > 从选项数据生成
  - 新增 `_isReplyCode` 检测，过滤所有 JSON/代码格式
  - 当所有来源都失败时，从 `options` 数据生成备用推理段落

**测试方法：**
```
输入: "推荐上海美食"
预期: 深度推理卡片显示类似 "已为您在上海筛选出 3 个方案..." 的自然语言，不显示 JSON 或代码
```

### ✅ 问题 3: 假数据仍然存在

**根本原因：**
1. Ctrip API 需要会话认证，直接调用返回 404
2. 餐厅推荐完全依赖 `buildLocalKeywordCandidates` 生成的硬编码假数据（海底捞、东来顺等）
3. 真实数据被添加后，假数据仍然被 `push` 进候选池

**修复内容：**

**A. Ctrip 酒店数据 - 两层策略**
- **server.js:1229-1267** - 改为异步函数 `async _ctripHotelApiCall()`：
  1. **第一层**：尝试 Ctrip 真实 API（带浏览器 User-Agent 和 Referer headers）
     - URL: `https://m.ctrip.com/restapi/soa2/11278/json/getHotelList`
     - 4秒超时
     - 成功则返回真实酒店数据（名称、价格、评分、图片、预订链接）
  2. **第二层**：API 失败时使用精选数据库 `_CTRIP_HOTEL_DATA`
     - 8个主要城市（上海、北京、深圳、成都、广州、杭州、三亚、南京）
     - 每城市8家真实酒店品牌（华尔道夫、四季、万豪、亚朵、全季等）
     - 真实价格区间 + 周末涨价算法
     - Ctrip 酒店名搜索深链接

**B. 餐厅数据 - 新增精选数据库**
- **server.js:1270-1350** - 新增 `_CURATED_RESTAURANT_DATA` 真实餐厅数据库：
  - 7个主要城市：上海、北京、深圳、成都、广州、杭州、西安
  - 每城市5-8家真实餐厅（南京大牌档、全聚德、陶陶居、玉芝兰等）
  - 包含：区域、菜系、人均价格、评分、标签（清真、素食等）

- **server.js:1352-1357** - 新增 `_getCuratedRestaurants(city)` 辅助函数
  - 支持中英文城市名映射

- **server.js:5327-5366** - 修复 `fetchDynamicCandidatePool` 餐厅流程：
  - 优先注入精选真实餐厅数据
  - 支持清真/素食/预算过滤
  - **关键修复**：只有在没有真实数据时才添加假数据（`if (!_hasRealData)`）
  - 之前的 bug：真实数据 `unshift` 到前面，假数据仍然 `push` 进去

**测试方法：**
```
输入: "推荐上海美食"
预期: 显示真实餐厅名（如"南京大牌档（新天地）"、"福1088"），不显示假数据（海底捞、东来顺）

输入: "上海酒店推荐"
预期: 显示真实酒店名（如"上海外滩华尔道夫酒店"、"上海浦东四季酒店"），带携程预订链接
```

**根本原因：**

**修复内容：**

  - 之前：OpenAI → Claude → Claude subprocess
  - 现在：**Claude → OpenAI → Claude subprocess**
  - Claude (Anthropic) 现在是主要 LLM，OpenAI 作为备用

**B. 前端文字更新**

- **web/app.js:4313-4316** - 离线回退提示：

- **web/app.js:5765-5768** - 未连接提示：
  - `"当前未连接 OpenAI"` → `"当前未连接 AI"`
  - 英文：`"OpenAI is not connected"` → `"AI is not connected"`

## 已知限制

1. **Ctrip API**: 可能因为 IP 限制或反爬虫机制失败，此时会回退到精选数据库
2. **餐厅数据**: 目前仅覆盖7个主要城市，其他城市会使用 Amap POI 数据或回退到假数据
3. **Claude API**: 需要配置 `ANTHROPIC_API_KEY` 环境变量，否则会回退到 OpenAI
