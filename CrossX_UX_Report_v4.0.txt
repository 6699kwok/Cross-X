ISSUE A: International SMS — Can it be solved without switching providers?
------------------------------------------------------------------------
Current state:
  - sms.js supports Tencent Cloud and Aliyun only
  - Both are Chinese domestic SMS platforms
  - Neither delivers reliably to AU/US/UK/EU numbers (requires special
    international tier, high cost, carrier rejections common)

Alternative approaches — RANKED:

  [Option 1 — RECOMMENDED] Add Twilio adapter to sms.js (pluggable)
    The SMS provider is already abstracted behind SMS_PROVIDER env var.
    Adding SMS_PROVIDER=twilio requires:
      - ~30 lines in sms.js: POST to api.twilio.com/2010-04-01/Accounts/
        {TWILIO_ACCOUNT_SID}/Messages.json with TWILIO_AUTH_TOKEN
      - No changes to user_auth.js or any route
    Cost: ~$0.0079/SMS globally. Works in 180+ countries.
    Verdict: FEASIBLE. Low effort, high impact. Should be built.

  [Option 2 — SHORT-TERM] Email OTP as fallback for international numbers
    If phone prefix is non-CN (not +86), offer "Send code to email instead"
    No external SMS dependency for foreign users.
    Requires: email field in login modal + nodemailer/SES integration
    Verdict: FEASIBLE. Medium effort. Good interim solution.

  [Option 3 — CURRENT WORKAROUND] Magic link via Google OAuth
    Google OAuth already implemented (AuthController.js:208)
    Foreign users can skip SMS entirely via "Continue with Google"
    Only works when GOOGLE_CLIENT_ID + GOOGLE_CLIENT_SECRET are set
    Verdict: ALREADY EXISTS. Zero code cost. Activate via env vars.

  Conclusion: International SMS CAN be solved. Priority order:
    1. Activate Google OAuth (zero effort — just needs env vars)
    2. Add Twilio adapter (~2h dev work)
    3. Email OTP fallback (~4h dev work)

ISSUE B: Mock Fulfillment (#13) — Can fake bookings be replaced?
------------------------------------------------------------------------
Current state:
  merchants.js createHotelBooking() falls back to:
    bookingId: "PHUB_MOCK_1a2b3c..."
    confirmationNo: "CX12345678"
    source: "mock"
  These are fake — no real reservation is made anywhere.

Alternative approaches — RANKED:

  [Option 1 — RECOMMENDED] Replace confirm button with deep-link redirect
    Instead of generating a fake CX confirmation number, redirect user to:
    - Booking.com pre-filled URL for the specific hotel
    - Agoda pre-filled URL
    - Klook for experiences/tours
    CrossX acts as travel planning + discovery layer, not booking engine.
    This is ALREADY DONE for the hotel card buttons in _buildBookBtnHtml()
    — just needs to be applied consistently to the "Book Now" checkout flow.
    Verdict: FEASIBLE, QUICK. 1-2h refactor of the booking confirmation flow.

  [Option 2] Connect PARTNER_HUB_BASE_URL (real API partner)
    merchants.js already has full Partner Hub integration code.
    When PARTNER_HUB_BASE_URL + PARTNER_HUB_KEY are set, real bookings work.
    Requires: signing an API agreement with a GDS (e.g., Hotelbeds, RateHawk,
              TravelGate, or Ctrip OpenPlatform)
    Verdict: PROPER SOLUTION but requires business partnership (weeks/months).

  [Option 3 — SHORT-TERM] Honest UX: show deep-link, be transparent
    Label the "Book" action as "Reserve on Booking.com" with external icon.
    User understands CrossX is a planner, not a booking engine.
    No fake confirmation numbers. Trust preserved.
    Verdict: IMMEDIATE FIX. No code changes needed beyond copy/label.

  Conclusion: Mock fulfillment CAN be replaced immediately via deep-links.
    The architecture supports it. Option 1 + Option 3 together = honest,
    functional booking experience within days, not months.

ISSUE C: "Foreign Card" Near Me Filter (#12) — Can data be populated?
------------------------------------------------------------------------
Current state:
  web/app.js line ~10937: filter removes item "n4" when foreignCard filter
  is active. Only one item is affected — most Near Me data lacks
  supportsForeignCard metadata.

Alternative approaches — RANKED:

  [Option 1 — RECOMMENDED] Populate via payment rail metadata
    CrossX already has a PaymentRailManager (lib/payments/rail.js) with
    supportsForeignCard: true/false per venue type.
    Map Near Me venue categories to rail capabilities at render time.
    e.g., "Foreign-friendly" venues = those accepting Stripe/Visa.
    Verdict: FEASIBLE. ~3h work. Data-driven, no manual curation needed.

  [Option 2] Hide filter until real data exists
    Remove the "Foreign Card" filter chip from Near Me UI.
    Avoids user confusion from a filter that barely works.
    Verdict: SAFEST SHORT-TERM. 15 min. Prevents trust damage.

  [Option 3] Curate a starter dataset of foreign-card venues per city
    Partner with popular international chains (Starbucks, McDonalds,
    international hotel chains) — all accept foreign cards.
    Tag them in Near Me data manually.
    Verdict: MEDIUM EFFORT. Requires content work but is accurate.

  Conclusion: Hiding the filter (Option 2) takes 15 minutes and prevents
    misleading UX. Option 1 provides a scalable solution alongside it.

PART 5 — PRODUCT STRATEGY ASSESSMENT
================================================================================

CURRENT PRODUCT POSITION:
  CrossX is a world-class AI travel planner for China, with genuinely impressive
  cultural intelligence, multilingual capability, and payment rail breadth.
  The planning experience is a real competitive advantage.

CORE IDENTITY QUESTION:
  CrossX must decide: Is it a BOOKING PLATFORM or a PLANNING + DISCOVERY PLATFORM?

  Option A — Booking Platform:
    Requires: Real hotel/tour API partnerships (GDS, Partner Hub, Ctrip API).
    Timeline: 3-6 months minimum to sign and integrate.
    Risk: Competing with Booking.com, Agoda, Ctrip — enormous moat to overcome.

  Option B — Planning + Discovery Platform (RECOMMENDED):
    CrossX is the "AI travel brain" that connects to best-in-class booking platforms.
    Deep-link architecture already built and working (_buildBookBtnHtml).
    Position: "Plan with AI. Book where you trust."
    Monetization: Affiliate commission from Booking.com/Agoda/Klook (6-12% per booking).
    Timeline: Remove fake confirmations + add affiliate IDs = 1-2 days work.
    Risk: Very low. Users already trust Booking.com for the actual booking.

  Affiliate revenue estimate:
    If CrossX refers 100 hotel bookings/month averaging $150/booking:
    $150 × 100 × 8% = $1,200/month from hotel referrals alone.
    Scales linearly with user growth. No fulfillment liability.

THE BOOKING TRUST GAP — ROOT CAUSE:
  The current fake confirmation flow tries to make CrossX look like a booking
  platform before it's actually one. This is worse than being honest about
  being a planning platform. Fake confirmations:
    - Destroy trust when users discover them (all 6 personas affected)
    - Create false expectation of real reservation
    - Risk real-world trip failures (no actual hotel room reserved)

  SOLUTION: Embrace the planning identity. Remove fake confirmations.
  Add honest, clear deep-links with affiliate tracking.
  This is both the ethical and the commercially superior path.
